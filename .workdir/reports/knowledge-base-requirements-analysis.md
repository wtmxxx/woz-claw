# RAG知识库系统需求分析报告

## 1. 项目背景

本项目基于 `rag-demo` 演示系统，旨在构建一个面向校园场景的知识库系统。系统通过文档切分（chunking）技术，将原始文档转化为适合检索增强生成（RAG）的结构化数据块，为后续的问答、检索和知识管理提供基础支撑。

## 2. 系统目标

- 实现文档的智能切分与结构化处理
- 支持多种文档格式（.txt、.md）
- 提供可视化切分效果展示
- 保留文档结构信息（标题层级、列表、表格）
- 支持参数化切分策略调整

## 3. 核心功能需求

### 3.1 文档上传与解析

- 支持上传 `.txt` 和 `.md` 格式文件
- 自动识别文件编码（UTF-8 / GBK）
- 空文件与无效文件校验

### 3.2 文档结构识别

- 识别标题层级（Markdown `#`、中文编号 `一、二、...`、数字编号 `1.2.3`）
- 排除误判（日期、时间、数据行）
- 支持多层级嵌套结构

### 3.3 智能切分策略

| 切分策略 | 触发条件 | 说明 |
|---------|---------|------|
| 章节优先 | 内容长度 ≤ max_size | 小章节整体保留，不切分 |
| 递归切分 | 内容长度 > max_size | 超长章节继续细分 |
| 表格切分 | 检测到表格块 | 按行切分并复制表头 |
| 列表保护 | 前导句+列表结构 | 保持前导句与列表项绑定 |

### 3.4 参数配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `base_max_size` | 1024 | 基础切分长度阈值（token） |
| `key_content_max_size` | 1536 | 重点内容切分长度阈值（token） |
| `overlap` | 100 | 句子级重叠长度（token） |

### 3.5 元数据生成

每个 chunk 需包含以下元数据：

- `chunk_id`：唯一标识符
- `section` / `section_path`：所属章节路径
- `type`：`text` 或 `table`
- `start_char` / `end_char`：在原文中的字符位置
- `prev_chunk_id` / `next_chunk_id`：前后 chunk 关系
- `is_key_content`：是否为重点内容
- `weight`：权重（重点内容为 1.5，普通内容为 1.0）
- `token_len`：token 长度

### 3.6 文档结构评分

系统自动评估文档结构质量：

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| 标题数量 | 10分/个（上限50） | 标题越多，结构越清晰 |
| 标题深度 | 5分/层（上限20） | 层级越深，结构越丰富 |
| 结构多样性 | 列表/表格加分（上限20） | 结构化内容加分 |

**评分等级：**

- `high`：60-100 分，结构清晰，适合语义分割
- `medium`：35-59 分，结构尚可
- `low`：0-34 分，结构偏弱

### 3.7 可视化展示

- 左侧：原文高亮显示（12种颜色循环）
- 右侧：分片列表（彩色序号标签）
- 交互：点击高亮 ↔ 自动滚动同步

## 4. 非功能需求

### 4.1 性能需求

- 单次处理文档大小：≤ 100KB
- 切分响应时间：≤ 2秒（常规文档）
- 支持并发用户：≥ 10

### 4.2 兼容性需求

- 浏览器：Chrome、Edge、Firefox 最新版
- 操作系统：Windows、macOS、Linux
- 文件编码：UTF-8、GBK

### 4.3 可维护性需求

- 代码模块化：`app/chunker.py`、`app/main.py`
- 配置分离：参数可通过 API 或界面调整
- 日志记录：关键操作需记录日志

## 5. 技术架构

### 5.1 后端技术栈

- 框架：FastAPI
- 模板引擎：Jinja2
- Token 计数：tiktoken（cl100k_base）
- 虚拟环境：uv（兼容 pip）

### 5.2 前端技术栈

- HTML5 + CSS3
- JavaScript（原生，无框架依赖）
- 交互库：无外部依赖

### 5.3 项目结构

```
rag-demo/

│   └── main.py         # FastAPI 路由

```

## 6. 典型场景分析

### 6.1 小型文档（< max_size）

**示例：** `club_event.txt`（校园摄影社通知）

- 切分结果：1 个 chunk
- 文档结构评分：低（标题较少）
- 处理策略：章节优先，整体保留

### 6.2 中型文档（≈ max_size）

**示例：** `campus_notice.md`（校园网升级通知）

- 切分结果：2-3 个 chunks
- 文档结构评分：中等
- 处理策略：章节优先 + 表格切分

### 6.3 大型文档（> max_size）

**示例：** `graduate_research_guidelines.md`（研究生科研指南）

- 切分结果：10+ 个 chunks
- 文档结构评分：高
- 处理策略：章节优先 + 递归切分 + 表格切分 + overlap

### 6.4 表格密集文档

**示例：** `long_table_split.md`（超长表格测试）

- 切分结果：多 chunks（表格按行切分）
- 文档结构评分：中等
- 处理策略：表格切分 + 表头复制

## 7. 需求优先级

| 优先级 | 功能 | 说明 |
|-------|------|------|
| P0 | 文档上传与解析 | 基础功能，必须实现 |
| P0 | 标题识别与切分 | 核心功能，必须实现 |
| P1 | 元数据生成 | 支持后续检索 |
| P1 | 可视化展示 | 提升用户体验 |
| P2 | 文档结构评分 | 辅助决策 |
| P2 | 参数化配置 | 提高灵活性 |
| P3 | 高级功能（语义切分） | 后续扩展 |

## 8. 风险与限制

### 8.1 技术风险

- Token 计数依赖 tiktoken，若环境缺失会降级为字符计数
- 表格识别规则可能误判复杂表格
- 中文标题格式多样，可能存在漏判

### 8.2 业务限制

- 当前仅支持文本文件，不支持 PDF、Word
- 切分结果依赖参数配置，需人工调优
- 无向量检索功能（需后续集成）

## 9. 后续扩展方向

- PDF 版面信息融合（字体/坐标）
- Embedding 语义断点切分
- 接入向量库并演示检索效果
- 支持更多文档格式（PDF、DOCX）
- 多语言支持（英文、日文等）

### 10.2 API 接口

**端点：** `POST /api/chunk`

**请求参数：**

- `file`：上传文件（multipart/form-data）
- `base_max_size`：基础切分长度（默认 1024）
- `key_content_max_size`：重点内容切分长度（默认 1536）
- `overlap`：重叠长度（默认 100）

```json
      "end_char": 56,

  ]

```
