# RAG知识库系统需求分析报告

## 1. 项目背景

这个项目基于 `rag-demo` 演示系统，目标是构建一个面向校园场景的知识库系统。系统通过文档切分技术，把原始文档转成适合检索增强生成的结构化数据块，为后续的问答、检索和知识管理打好基础。

我看了现有的演示系统，发现它已经能处理一些基本文档，但校园场景的需求更复杂。校园文档种类多，从简单的活动通知到复杂的科研指南，格式也五花八门。现有的切分逻辑需要针对这些特点做调整。

## 2. 系统目标

- 实现文档的智能切分与结构化处理
- 支持多种文档格式（.txt、.md）
- 提供可视化切分效果展示
- 保留文档结构信息（标题层级、列表、表格）
- 支持参数化切分策略调整

除了这些技术目标，我觉得系统还应该考虑实际使用场景。比如，教务处老师上传通知时，希望快速看到切分效果；学生查询资料时，需要准确的检索结果。系统设计要兼顾不同用户的需求。

## 3. 核心功能需求

### 3.1 文档上传与解析

- 支持上传 `.txt` 和 `.md` 格式文件
- 自动识别文件编码（UTF-8 / GBK）
- 空文件与无效文件校验

实际测试中发现，有些校园文档用的是 GBK 编码，特别是从旧系统导出的文件。编码识别失败会导致乱码，所以这个功能很重要。另外，文件大小也要有限制，避免大文件拖慢系统。

### 3.2 文档结构识别

- 识别标题层级（Markdown `#`、中文编号 `一、二、...`、数字编号 `1.2.3`）
- 排除误判（日期、时间、数据行）
- 支持多层级嵌套结构

校园文档的标题格式比较随意。有的用"第一章"，有的用"1.1"，还有的直接用"一、活动安排"。识别逻辑要能处理这些变体。测试时发现，日期如"2024年10月5日"容易被误判为标题，需要特别排除。

### 3.3 智能切分策略

| 切分策略 | 触发条件 | 说明 |
|---------|---------|------|
| 章节优先 | 内容长度 ≤ max_size | 小章节整体保留，不切分 |
| 递归切分 | 内容长度 > max_size | 超长章节继续细分 |
| 表格切分 | 检测到表格块 | 按行切分并复制表头 |
| 列表保护 | 前导句+列表结构 | 保持前导句与列表项绑定 |

这些策略是在测试中总结出来的。比如"章节优先"策略，是因为发现把完整的小章节切开会破坏语义连贯性。"表格切分"策略则是为了解决超长表格的问题——复制表头能让每个切分块都能独立理解。

### 3.4 参数配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `base_max_size` | 1024 | 基础切分长度阈值（token） |
| `key_content_max_size` | 1536 | 重点内容切分长度阈值（token） |
| `overlap` | 100 | 句子级重叠长度（token） |

参数默认值是通过测试文档调整出来的。1024 token 大约对应 700-800 个中文字符，这个长度在测试中表现平衡。重点内容（如定义、结论）可以适当放宽到 1536 token，保留更多上下文。重叠 100 token 能避免切分边界丢失关键信息。

### 3.5 元数据生成

每个 chunk 需要包含以下元数据：

- `chunk_id`：唯一标识符
- `section` / `section_path`：所属章节路径
- `type`：`text` 或 `table`
- `start_char` / `end_char`：在原文中的字符位置
- `prev_chunk_id` / `next_chunk_id`：前后 chunk 关系
- `is_key_content`：是否为重点内容
- `weight`：权重（重点内容为 1.5，普通内容为 1.0）
- `token_len`：token 长度

元数据设计考虑了后续的检索需求。比如 `section_path` 记录了章节层级，检索时可以优先匹配同章节内容。`weight` 字段让重点内容在检索结果中排名更高。这些设计都是为了让检索更准确。

### 3.6 文档结构评分

系统自动评估文档结构质量：

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| 标题数量 | 10分/个（上限50） | 标题越多，结构越清晰 |
| 标题深度 | 5分/层（上限20） | 层级越深，结构越丰富 |
| 结构多样性 | 列表/表格加分（上限20） | 结构化内容加分 |

**评分等级：**

- `high`：60-100 分，结构清晰，适合语义分割
- `medium`：35-59 分，结构尚可
- `low`：0-34 分，结构偏弱

评分系统是在分析测试文档后设计的。结构清晰的文档（如科研指南）得分高，切分效果也好；结构松散的文档（如简单通知）得分低，可能需要不同的处理策略。这个评分能帮助用户了解文档质量。

### 3.7 可视化展示

- 左侧：原文高亮显示（12种颜色循环）
- 右侧：分片列表（彩色序号标签）
- 交互：点击高亮 ↔ 自动滚动同步

可视化设计考虑了用户体验。用不同颜色区分切分块，用户一眼就能看出切分效果。点击交互让用户能快速定位原文位置。测试时发现，这个功能对调参很有帮助——能看到参数调整后的实时效果。

## 4. 非功能需求

### 4.1 性能需求

- 单次处理文档大小：≤ 100KB
- 切分响应时间：≤ 2秒（常规文档）
- 支持并发用户：≥ 10

性能指标是根据校园场景估算的。100KB 文档大约 5-6 万字，足够覆盖大多数校园文档。2秒响应时间能保证用户体验，不会让用户等太久。并发 10 用户能满足小型团队同时使用。

### 4.2 兼容性需求

- 浏览器：Chrome、Edge、Firefox 最新版
- 操作系统：Windows、macOS、Linux
- 文件编码：UTF-8、GBK

兼容性要求基于校园实际情况。校园里电脑系统多样，浏览器版本也不统一。文件编码要支持 UTF-8 和 GBK，因为有些旧文档还是 GBK 编码。

### 4.3 可维护性需求

- 代码模块化：`app/chunker.py`、`app/main.py`
- 配置分离：参数可通过 API 或界面调整
- 日志记录：关键操作需记录日志

模块化设计让后续维护更容易。切分逻辑独立在 `chunker.py` 中，需要调整时不用动其他部分。配置分离让非技术人员也能调整参数。日志记录帮助排查问题，特别是处理复杂文档时。

## 5. 技术架构

### 5.1 后端技术栈

- 框架：FastAPI
- 模板引擎：Jinja2
- Token 计数：tiktoken（cl100k_base）
- 虚拟环境：uv（兼容 pip）

选择 FastAPI 是因为它轻量、性能好，适合这种计算密集型任务。Jinja2 模板引擎简单够用。tiktoken 是 OpenAI 的 token 计数工具，准确度高。uv 虚拟环境管理方便，部署简单。

### 5.2 前端技术栈

- HTML5 + CSS3
- JavaScript（原生，无框架依赖）
- 交互库：无外部依赖

前端用原生技术，不依赖框架。这样部署简单，加载快。校园网络环境复杂，减少外部依赖能提高稳定性。交互逻辑自己实现，虽然代码量稍多，但可控性更好。

### 5.3 项目结构

```
rag-demo/
├── app/
│   ├── chunker.py      # 核心切分逻辑
│   └── main.py         # FastAPI 路由
├── templates/
│   └── index.html      # 页面模板
├── static/
│   ├── style.css       # 样式文件
│   └── app.js          # 前端交互逻辑
├── sample/             # 测试文档
└── docs/               # 文档与改进记录
```

项目结构清晰，功能分离明确。`chunker.py` 专注切分算法，`main.py` 处理 Web 请求。静态资源分开存放，便于缓存。测试文档和文档记录也有专门目录。

## 6. 典型场景分析

### 6.1 小型文档（< max_size）

**示例：** `club_event.txt`（校园摄影社通知）

- 切分结果：1 个 chunk
- 文档结构评分：低（标题较少）
- 处理策略：章节优先，整体保留

这种文档简单，内容少。直接整体保留效果最好，切分了反而破坏完整性。结构评分低是正常的，不影响使用。

### 6.2 中型文档（≈ max_size）

**示例：** `campus_notice.md`（校园网升级通知）

- 切分结果：2-3 个 chunks
- 文档结构评分：中等
- 处理策略：章节优先 + 表格切分

这类文档有简单结构，可能包含表格。按章节切分能保持逻辑完整，表格单独处理保证可读性。

### 6.3 大型文档（> max_size）

**示例：** `graduate_research_guidelines.md`（研究生科研指南）

- 切分结果：10+ 个 chunks
- 文档结构评分：高
- 处理策略：章节优先 + 递归切分 + 表格切分 + overlap

大型文档结构复杂，章节多，内容深。递归切分处理超长章节，overlap 避免边界信息丢失。这种文档切分后检索效果最好。

### 6.4 表格密集文档

**示例：** `long_table_split.md`（超长表格测试）

- 切分结果：多 chunks（表格按行切分）
- 文档结构评分：中等
- 处理策略：表格切分 + 表头复制

表格文档特殊，按行切分时复制表头，让每个切分块都能独立理解。测试中发现，不复制表头的话，后面的行就不知道列名是什么。

## 7. 需求优先级

| 优先级 | 功能 | 说明 |
|-------|------|------|
| P0 | 文档上传与解析 | 基础功能，必须实现 |
| P0 | 标题识别与切分 | 核心功能，必须实现 |
| P1 | 元数据生成 | 支持后续检索 |
| P1 | 可视化展示 | 提升用户体验 |
| P2 | 文档结构评分 | 辅助决策 |
| P2 | 参数化配置 | 提高灵活性 |
| P3 | 高级功能（语义切分） | 后续扩展 |

优先级划分基于功能依赖关系。基础功能必须先实现，否则系统无法运行。元数据和可视化虽然重要，但可以稍后完善。高级功能如语义切分，需要更多研究和测试。

## 8. 风险与限制

### 8.1 技术风险

- Token 计数依赖 tiktoken，若环境缺失会降级为字符计数
- 表格识别规则可能误判复杂表格
- 中文标题格式多样，可能存在漏判

tiktoken 依赖是个风险，但准备了降级方案——用字符计数代替。表格识别规则在测试中表现不错，但复杂表格（如合并单元格）可能处理不好。中文标题识别已经覆盖常见格式，但总有特殊情况。

### 8.2 业务限制

- 当前仅支持文本文件，不支持 PDF、Word
- 切分结果依赖参数配置，需人工调优
- 无向量检索功能（需后续集成）

这些限制是当前版本的设计选择。先做好文本处理，再扩展其他格式。参数调优需要经验，但提供了默认值和可视化工具帮助调整。向量检索是后续计划，当前版本专注切分。

## 9. 后续扩展方向

- PDF 版面信息融合（字体/坐标）
- Embedding 语义断点切分
- 接入向量库并演示检索效果
- 支持更多文档格式（PDF、DOCX）
- 多语言支持（英文、日文等）

扩展方向考虑了实际需求。PDF 是校园常用格式，需要支持。语义切分能提高切分质量，但需要更多测试。向量检索是 RAG 系统的核心，必须集成。多语言支持为国际化校园准备。

### 10.2 API 接口

**端点：** `POST /api/chunk`

**请求参数：**

- `file`：上传文件（multipart/form-data）
- `base_max_size`：基础切分长度（默认 1024）
- `key_content_max_size`：重点内容切分长度（默认 1536）
- `overlap`：重叠长度（默认 100）

**响应格式：**

```json
{
  "doc_id": "doc_xxx",
  "summary": {
    "total_chunks": 14,
    "text_chunks": 12,
    "table_chunks": 2,
    "avg_len": 87.79,
    "doc_structure_score": 65,
    "structure_quality": "high",
    "split_strategy": "章节优先 / 递归切分 / 表格切分"
  },
  "chunks": [
    {
      "chunk_id": "doc_xxx_0000",
      "text": "...",
      "section": "一、活动时间",
      "section_path": "一、活动时间",
      "type": "text",
      "is_key_content": false,
      "token_len": 45,
      "start_char": 12,
      "end_char": 56,
      "prev_chunk_id": null,
      "next_chunk_id": "doc_xxx_0001",
      "weight": 1.0
    },
    ...
  ]
}
```

API 设计考虑了易用性。参数都有默认值，用户可以不传。响应包含摘要信息和详细切分结果。摘要帮助快速了解切分效果，详细结果供后续处理使用。

## 11. 实施建议

基于以上分析，我建议实施时注意以下几点：

1. **分阶段开发**：先实现 P0 功能，确保系统能跑起来，再逐步完善
2. **持续测试**：用测试文档反复验证，特别是边界情况
3. **用户反馈**：找实际用户试用，收集改进意见
4. **文档完善**：记录开发过程中的经验和问题，方便后续维护

校园知识库系统有实际需求，但实施时要考虑现实约束。技术方案要实用，不能过于复杂。用户体验要重视，特别是非技术人员的使用。

## 12. 总结

这个需求分析基于现有 `rag-demo` 系统，针对校园场景做了调整和扩展。系统设计考虑了实际使用需求，从文档处理到可视化展示都有考虑。

技术方案选择平衡了功能和复杂度，用成熟技术实现核心需求。风险控制有预案，扩展方向有规划。

实施建议强调渐进式开发，先保证基础功能稳定，再逐步完善。测试和用户反馈是关键，能帮助系统更好地满足实际需求。

总的来说，这个方案可行，能构建一个实用的校园知识库切分系统。后续可以根据使用情况继续优化和扩展。
